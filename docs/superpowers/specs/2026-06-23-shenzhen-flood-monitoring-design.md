# 深圳市积水监测与预测系统顶层设计

> 更新基线：2026-06-24  
> 设计口径：先完成实时水位监测闭环，再补坐标、规则研判和训练数据闭环；初期不称为模型预测，后期在历史样本足够后训练时序模型并接入预测服务。

## 1. 项目定位

本项目建设一套深圳市积水监测与预测系统。系统不是单纯地图展示页，而是围绕“水位数据接入、原始数据归档、数据治理、坐标补全、地图监测、规则研判、训练数据沉淀、模型预测”形成完整闭环。

系统需要回答：

- 哪里出现积水或积水风险。
- 当前水位、风险等级和趋势是什么。
- 数据是否及时、可信、可追溯。
- 点位坐标来自哪里，精度和审核状态如何。
- 在已有历史水位和空间特征基础上，未来短时积水风险如何变化。
- 后续模型训练、评估、版本发布和预测结果回写如何进入系统。

## 2. 建设阶段

### 2.1 第一阶段：监测闭环

先打通可运行的基础系统：

```text
实时/定时水位接入
-> 原始 JSON/CSV 归档
-> SQLite 入库
-> 数据质量检查
-> 坐标补全
-> FastAPI 接口
-> Vue + 高德地图展示
```

第一阶段必须完成：

- 开放数据采集和原始归档。
- SQLite 入库和导入批次记录。
- 空水位、重复 ID、时间口径检查。
- 高德 API 初步坐标补全。
- 坐标来源、精度状态、审核状态记录。
- 地图点位、历史曲线、统计面板、数据状态页面。

### 2.2 第二阶段：规则研判

初期没有训练模型，因此不称为模型预测。该阶段只做规则研判和趋势提示：

- 基于当前水位计算风险等级。
- 基于最近有效水位序列计算上升、下降、稳定、波动。
- 基于数据新鲜度给出规则说明。
- 将研判结果回写数据库并展示到前端。

规则研判输出：

| 字段 | 说明 |
|---|---|
| `risk_level` | normal、attention、warning、danger、nodata |
| `trend` | rising、falling、stable、fluctuating、nodata |
| `rule_version` | 规则版本 |
| `rule_explanation` | 规则说明 |
| `generated_at` | 生成时间 |

### 2.3 第三阶段：模型训练与预测

当历史水位、临时坐标和扩展数据沉淀足够后，再训练时序模型。

模型训练目标：

```text
过去一段时间的水位序列
+ 时间特征
+ 点位特征
+ 坐标/空间特征
+ 降水量特征，获取后接入
-> 预测未来水位或未来风险等级
```

当前尚未获取降水量数据，因此模型分为无降水基线和降水增强两个阶段。降水量到位前，先用水位时序做基线；降水量到位后，在 `/download/市水务局积涝水位数据` 的积涝水位数据基础上增加降水量维度，形成水位与降水之间的映射特征。

```text
基线阶段：基于历史水位的短时水位趋势/风险预测
增强阶段：接入降水量后的多源积水风险等级预测
```

降雨等级评估：

- 对 6 月数据按降雨等级划分场景，至少覆盖小雨、中雨、暴雨。
- 对每个降雨等级分别统计预测准确率，避免只看全量平均准确率掩盖暴雨场景效果。
- 前端展示时，降雨等级或预测风险等级颜色由浅到深表达强度增加。

模型预测输出：

| 字段 | 说明 |
|---|---|
| `target_time` | 预测目标时间 |
| `predicted_level_m` | 预测水位，可选 |
| `predicted_risk_level` | 预测风险等级 |
| `predicted_trend` | 预测趋势 |
| `confidence` | 置信度，仅模型阶段使用 |
| `model_version` | 模型版本 |
| `feature_version` | 特征版本 |
| `generated_at` | 生成时间 |

## 3. 坐标数据策略

当前测站基础信息缺少经纬度，坐标不能编造。采用两阶段策略：

```text
测站名称/地址/编码
-> 高德 API 初步坐标
-> 标记 coord_source=amap、coord_quality=provisional
-> 人工审核
-> 申请政府标准坐标
-> 标准坐标校正/替换
-> 样本修正和模型重训
```

坐标要求：

- 高德坐标可以用于初期地图落点和初期训练样本。
- 高德坐标必须保留来源和精度标记。
- 不能把高德坐标当作政府标准坐标。
- 政府标准坐标到位后，需要更新空间特征、重新评估模型，必要时重训。

## 4. 总体架构

系统采用前后端分离：

| 层级 | 技术/模块 | 说明 |
|---|---|---|
| 数据源 | 深圳开放数据、高德 API、政府标准坐标 | 水位、测站信息、坐标补全 |
| 数据采集 | Python 脚本，后续可接调度 | 下载、归档、增量导入 |
| 数据存储 | SQLite，后续 PostgreSQL + PostGIS | 当前快速验证，后续空间能力增强 |
| 数据治理 | FastAPI services/tasks | 质量检查、最新状态、风险趋势 |
| 坐标补全 | 高德地理编码 + 人工审核 + 政府标准坐标 | 坐标来源和精度可追溯 |
| 研判/预测服务 | 规则研判，后期模型预测 | 初期规则，后期模型 |
| 后端接口 | FastAPI | 地图、统计、历史、状态、坐标、研判/预测 |
| 前端应用 | Vue + 高德 JS API | 指挥地图、历史查询、点位校核、数据状态、预测结果 |

## 6. 训练集与测试集口径

时序模型不能随机划分数据，应按预测目标时间切分：

```text
训练集：较早时间段
验证集：中间时间段
测试集：最后时间段
```

当前固定切分可作为基线参考：

```text
训练集：2026-01-01 至 2026-04-30
验证集：2026-05-01 至 2026-05-31
测试集：2026-06-01 至 2026-06-23
```

正式评估采用 6 月滚动测试：

```text
6 月 1 日评估：使用 6 月 1 日之前的数据训练，6 月 1 日及之后作为测试集
6 月 2 日评估：使用 6 月 2 日之前的数据训练，6 月 2 日及之后作为测试集
...
6 月 N 日评估：使用 6 月 N 日之前的数据训练，6 月 N 日及之后作为测试集
```

注意：

- 当前没有降水量时，先运行无降水基线；降水量获取后必须纳入增强模型和同一套 6 月滚动测试。
- 滚动测试不得使用预测日当天及之后的数据训练，避免时间泄漏。
- 6 月测试结果需要按小雨、中雨、暴雨分层统计准确率。
- 高德临时坐标可以作为空间特征，但必须带 `coord_source` 和 `coord_quality`。
- 政府标准坐标到位后，应重新生成空间特征并复评模型。

## 7. API 范围

主线 API：

| API | 用途 |
|---|---|
| `GET /api/map/points` | 地图点位、最新水位、风险等级、趋势、坐标来源 |
| `GET /api/points/{id}/history` | 单点历史水位 |
| `GET /api/stats/overview` | 全市风险统计 |
| `GET /api/imports/latest` | 最近导入状态 |
| `POST /api/locations/geocode-candidates` | 高德候选坐标 |
| `POST /api/locations/{id}/review` | 坐标审核 |
| `GET /api/assessments` | 规则研判结果 |
| `GET /api/predictions` | 模型阶段预测结果 |
