## 1. Day 2 结论

Day 2 的核心任务是把 Day 1 已确认的一期基础闭环和二期扩展目标，转化为可以指导后续工程落地的顶层设计。

今日确认的总体路线如下：

```text
需求范围 -> 系统架构 -> 数据模型 -> 后端 API -> 前端页面 -> 主链路 -> 降级规则 -> Day 3 工程准备
```

本项目不是单纯制作一个地图页面，而是建设面向深圳城市防汛和应急指挥的积水实时监测系统。系统需要先完成一期可运行闭环，再在此基础上扩展二期指挥增强能力。

Day 2 的结论是：

- 一期以 SQLite 快速打通数据采集、入库、坐标校核、风险计算、地图展示、统计和历史查询。
- 后续增强在主线基础上迁移到 PostgreSQL + PostGIS，并补充调度、点位级阈值和扩展图层；告警、权限审计和工单只保留设计位置。
- 前端采用 Vue，默认首页为高德地图指挥页面，同时保留后台管理页面。
- 后端采用 FastAPI，当前优先实现地图、统计、历史、校核、预测和导入状态接口；告警、工单和审计接口后置。
- 地图能力通过 `MapProvider` 封装，避免业务组件直接依赖高德 SDK。
- 所有地图点位必须使用审核通过坐标，不能凭空补经纬度。
- 空水位、重复记录、接口失败、PostGIS 未就绪、扩展图层缺数据等情况都需要有明确降级规则。

## 4. 系统定位

系统服务对象是城市防汛、应急指挥和运维管理人员。

系统需要回答以下问题：

- 深圳哪里正在发生积水或存在积水风险。
- 当前点位水位是多少。
- 当前风险等级是什么。
- 水位趋势是上升、下降、稳定还是波动。
- 高风险点位集中在哪些行政区。
- 数据是否及时，是否存在导入失败或延迟。
- 点位坐标是否可信，是否经过人工审核。
- 预测模型如何辅助未来积水风险研判。
- 后续是否需要告警和工单辅助处置。
- 降雨、道路、历史易涝点是否可以作为后续图层辅助研判。

因此系统应设计为数据驱动的防汛指挥系统，而不是静态地图展示页。

## 5. 总体技术路线

| 模块 | 技术选择 | Day 2 确认结论 |
|---|---|---|
| 后端 | FastAPI | 优先负责地图、统计、历史、校核、预测、数据状态 API；调度、审计等后置 |
| 前端 | Vue | 优先负责指挥地图、历史查询、点位校核、数据状态页面 |
| 地图 | 高德 JS API | 一期默认地图底座，地图能力通过 `MapProvider` 封装 |
| 一期数据库 | SQLite | 快速验证现有数据链路和一期闭环 |
| 后续数据库 | PostgreSQL + PostGIS | 后续支撑空间点位、行政区边界、道路、易涝点、降雨和空间查询 |
| 数据采集 | Python 脚本 | 先复用现有脚本，后续再纳入后端调度任务 |
| 原始数据 | JSON/CSV 归档 | 保留开放数据原始分页和汇总文件，支持追溯 |
| 风险计算 | 规则计算 | 主线使用统一阈值，点位级阈值后置 |
| 预测模型 | 规则型或轻量预测 | 主线支撑能力，先展示未来风险等级或趋势，不训练复杂模型 |

## 6. 总体架构分层

系统分为五层。

### 6.1 数据源层

数据源层负责提供系统所需的原始数据。

一期核心数据源：

- 市水务局积涝点水位数据。
- 市水务局测站基本信息表。
- 本地已下载 CSV 文件。
- 本地 SQLite 数据库中的历史数据。

一期辅助数据源：

- 深圳行政区边界 GeoJSON。
- 高德地理编码服务。

后续扩展数据源：

- 降雨数据。
- 道路数据。
- 历史易涝点数据。
- 预测结果样例数据。

当前已确认的关键问题是：测站基础信息表缺少经纬度。因此地图点位不能直接展示全部测站，必须先生成候选坐标，再由人工审核后进入地图。

### 6.2 数据采集与处理层

数据采集与处理层负责把开放数据转成可入库、可计算、可展示的数据。

一期职责：

- 拉取开放数据接口。
- 保存原始分页 JSON。
- 汇总 CSV。
- 清洗字段。
- 导入 SQLite。
- 处理空水位。
- 处理重复记录。
- 生成最新点位状态。
- 计算风险等级。
- 计算简单趋势。

后续增强职责：

- 将采集脚本纳入后端调度任务。
- 记录每次同步任务的开始时间、结束时间、成功失败状态和失败原因。
- 将一期数据迁移到 PostgreSQL + PostGIS。
- 支持调度任务手动触发和定时执行。
- 在数据更新后触发风险重算、最新状态更新和轻量预测；告警生成后置。

### 6.3 数据存储层

一期使用 SQLite，目标是快速打通基础闭环。

SQLite 适合一期原因：

- 已经存在本地数据库。
- 便于快速验证字段、导入脚本和查询逻辑。
- 不依赖复杂数据库环境。
- 适合 Day 1 到 Day 12 的一期演示闭环。

后续可迁移到 PostgreSQL + PostGIS。

PostGIS 必要性：

- 存储审核通过的点位空间坐标。
- 存储行政区边界。
- 存储道路图层。
- 存储历史易涝点图层。
- 支持空间索引和空间查询。
- 支持后续按行政区、道路附近、风险区域等维度分析。

### 6.4 服务层

服务层采用 FastAPI。

服务层职责：

- 对外提供 REST API。
- 封装业务逻辑。
- 封装数据库访问。
- 执行风险计算。
- 执行趋势计算。
- 后续生成告警、推进工单状态、记录审计日志。
- 管理同步任务。

服务层不应让前端直接访问数据库，也不应让前端直接调用采集脚本。

### 6.5 前端应用层

前端采用 Vue。

前端职责：

- 展示指挥地图首页。
- 展示点位 Marker、行政区边界和风险颜色。
- 展示全市和行政区统计。
- 展示点位详情。
- 查询历史水位。
- 审核点位坐标。
- 查看数据状态。
- 主线展示预测结果；后续展示告警、阈值、工单、审计和图层能力。

地图能力通过 `MapProvider` 封装，业务页面不直接散落调用高德 SDK。

## 7. 一期闭环详细设计

一期目标是完成基础可运行闭环。

一期主链路：

```text
开放数据采集 -> 原始数据归档 -> SQLite 入库 -> 数据质量检查 -> 坐标候选 -> 人工审核 -> 风险计算 -> 高德地图展示 -> 统计与历史查询
```

### 7.1 开放数据采集

采集对象：

- 积涝点水位数据。
- 测站基本信息表。
- 必要时补充水库水位表作为数据参考。

采集结果：

- 原始分页 JSON。
- 汇总 CSV。
- 下载元数据。
- 数据下载流程说明。

设计要求：

- 原始数据必须保留，便于追溯。
- 汇总数据必须保留来源路径。
- 下载失败要记录失败原因。
- 后续导入不能覆盖原始数据。

### 7.2 SQLite 入库

SQLite 入库用于快速形成一期数据底座。

入库对象：

- 测站基础信息。
- 积涝点水位观测。
- 导入批次。
- 数据源状态。

入库要求：

- 保留原始水位文本。
- 将可解析水位转成数值字段。
- 空水位保留，但业务计算时排除。
- 原始记录 ID 完全重复时不重复进入业务统计。
- 每次导入记录批次状态、时间、行数和错误信息。

### 7.3 坐标候选与人工审核

由于测站基础信息缺少经纬度，一期必须设计坐标校核流程。

流程：

```text
点位名称/地址 -> 高德地理编码候选 -> 候选坐标列表 -> 人工审核 -> 审核通过坐标 -> 地图展示
```

设计规则：

- 候选坐标可以来自高德地理编码。
- 人工可以确认、驳回或手动修正坐标。
- 只有审核通过坐标的点位进入地图。
- 未审核点位保留在校核列表。
- 不能为了地图完整性编造经纬度。

### 7.4 风险等级计算

一期使用统一风险规则。

风险等级：

- `normal`：正常。
- `attention`：关注。
- `warning`：警戒。
- `danger`：危险。
- `nodata`：无数据。

一期风险计算要求：

- 只使用有效水位数值参与计算。
- `water_level_m IS NULL` 的记录不参与风险判断。
- 无有效水位时显示 `nodata`。
- 风险等级需要可解释，点位详情中应展示触发依据。

### 7.5 趋势计算

一期先做简单趋势，并在此基础上实现规则型或轻量预测，不做复杂机器学习训练。

趋势类型：

- `rising`：持续上升。
- `falling`：持续下降。
- `stable`：基本稳定。
- `fluctuating`：波动。
- `nodata`：数据不足。

趋势计算可以基于最近若干条有效观测值。

### 7.6 高德地图展示

地图展示内容：

- 深圳地图底图。
- 行政区边界。
- 审核通过的积涝点 Marker。
- 风险颜色。
- 点位弹窗。
- 高风险点位列表。
- 全市态势统计。

Marker 风险颜色：

- 正常：绿色。
- 关注：黄色。
- 警戒：橙色。
- 危险：红色。
- 无数据：灰色。

点位弹窗展示：

- 点位名称。
- 当前水位。
- 风险等级。
- 趋势。
- 观测时间。
- 所属行政区。
- 历史曲线入口。

### 7.7 统计与历史查询

一期统计包括：

- 点位总数。
- 正常数量。
- 关注数量。
- 警戒数量。
- 危险数量。
- 无数据数量。
- 最新更新时间。
- 各行政区风险统计。

历史查询包括：

- 单点历史水位曲线。
- 历史明细表。
- 时间范围筛选。
- 点位筛选。

### 7.8 数据状态查看

数据状态页用于回答“数据是否可信、是否更新成功”。

展示内容：

- 数据源名称。
- 最近一次拉取时间。
- 拉取行数。
- 导入行数。
- 成功或失败状态。
- 错误信息。
- 原始文件路径。

## 8. 后置扩展详细设计

后置扩展目标是在主线闭环稳定后，逐步增强防汛指挥能力。当前阶段只保留设计，不要求先完成告警中心、用户角色审计和处置工单。预测模型不属于该后置范围。

后置增强链路：

```text
PostGIS 空间底座 -> 后端调度 -> 点位级阈值 -> 扩展图层 -> 告警/工单/审计预留
```

### 8.1 PostgreSQL + PostGIS

后续可将 SQLite 数据底座迁移到 PostgreSQL + PostGIS。该能力不阻塞当前主线交付。

迁移目标：

- 迁移监测点。
- 迁移水位历史。
- 迁移审核通过坐标。
- 建立空间字段。
- 建立空间索引。
- 保留一期数据质量口径。

PostGIS 扩展对象：

- 行政区边界。
- 道路图层。
- 历史易涝点。
- 降雨点位或网格。
- 点位坐标。

### 8.2 后端调度任务

后续可将采集脚本纳入后端调度体系。当前主线先保证脚本可手动运行、结果可入库、状态可追溯。

调度能力：

- 手动触发。
- 定时执行。
- 任务启停。
- 最近运行状态。
- 成功失败记录。
- 失败原因查看。

调度任务完成后，应触发：

- 数据入库。
- 数据质量检查。
- 最新状态更新。
- 风险重算。
- 告警生成。

### 8.3 点位级阈值

后续支持点位级风险阈值。当前主线先使用统一风险规则。

规则：

- 点位配置了专属阈值时，优先使用点位级阈值。
- 点位没有专属阈值时，回退全局阈值。
- 阈值更新后，需要重算对应点位风险。
- 阈值变更审计后置。

### 8.4 告警中心

告警中心是后置分支能力。当前阶段不先实现，只保留触发条件、状态和接口设计。

告警来源：

- 水位达到阈值。
- 数据延迟。
- 数据缺失。
- 后续可扩展为降雨风险、预测风险。

告警状态：

- `active`：待处理。
- `acknowledged`：已确认。
- `closed`：已关闭。

告警功能：

- 告警列表。
- 告警详情。
- 按等级筛选。
- 按状态筛选。
- 按行政区筛选。
- 告警确认。
- 从告警创建工单。

告警生成要求：

- 同一告警条件已有 active 告警时，不重复创建。
- 风险解除后可以关闭告警或标记为已恢复。
- 告警确认和关闭需要记录审计日志。

### 8.5 用户、角色和审计

用户、角色和审计是后置分支能力。当前阶段不先实现，不阻塞主线页面和接口。

用户角色目标：

- 支持后台登录。
- 支持用户列表。
- 支持角色列表。
- 支持页面保护。
- 支持操作审计。

建议角色：

- 管理员：可配置阈值、查看审计、管理调度。
- 指挥人员：可查看地图、告警和工单。
- 处置人员：可查看和处理工单。
- 只读用户：只查看地图和统计。

审计日志记录对象：

- 登录。
- 阈值修改。
- 告警确认。
- 工单创建。
- 工单状态变更。
- 手动触发同步。
- 坐标审核。

### 8.6 降雨图层

降雨图层用于辅助分析积水原因和风险。

展示形式：

- 降雨点位。
- 降雨数值。
- 降雨等级颜色。
- 观测时间。

一期不强依赖降雨数据。二期如果真实数据不稳定，可以先使用可信样例数据完成图层接口和前端展示。

### 8.7 道路图层

道路图层用于展示积水点与道路之间的空间关系。

展示形式：

- 道路线。
- 道路名称。
- 所属行政区。
- 道路等级。

本轮不做道路通行能力精细分析，只做图层展示和后续扩展预留。

### 8.8 历史易涝点图层

历史易涝点图层用于辅助识别重点风险区域。

展示内容：

- 易涝点名称。
- 所属行政区。
- 风险说明。
- 数据来源。
- 地图点或面。

历史易涝点数据源不稳定时，可以用样例 GeoJSON 或 CSV 支撑演示。

### 8.9 处置工单

工单用于形成告警后的处置闭环，属于后置分支。当前阶段只保留设计。

状态流转：

```text
new -> assigned -> processing -> completed -> closed
```

工单字段：

- 工单标题。
- 关联告警。
- 关联点位。
- 处理人。
- 状态。
- 创建时间。
- 完成时间。

工单功能：

- 从告警创建工单。
- 工单列表。
- 工单详情。
- 分派处理人。
- 推进状态。
- 关闭工单。

### 8.10 主线预测结果展示

预测模型属于主线支撑能力。本轮不训练复杂机器学习模型，但需要先实现规则型或轻量预测接口和页面。

预测展示内容：

- 点位名称。
- 当前风险等级。
- 预测目标时间。
- 预测风险等级。
- 预测趋势。
- 置信度。
- 规则或模型版本。
- 生成时间。

预测结果可以展示在：

- 点位详情。
- 预测结果页面。

预测结果必须标明生成时间和版本，避免被误解为实时权威预报。

## 13. 主链路设计

### 13.1 数据同步链路

```text
开放数据接口 -> 原始 JSON/CSV -> 导入数据库 -> 数据质量检查 -> 最新状态快照 -> 风险与趋势计算 -> 地图和统计 API
```

### 13.2 坐标校核链路

```text
点位基础信息 -> 高德地理编码候选 -> 人工审核 -> 审核通过坐标 -> 地图展示
```

### 13.3 风险计算链路

```text
有效水位 -> 阈值匹配 -> 风险等级 -> 最新状态 -> 地图 Marker 和统计面板
```

### 13.4 后置告警处置链路

```text
最新状态 -> 阈值触发 -> 告警生成 -> 告警确认 -> 创建工单 -> 工单流转 -> 工单关闭
```

### 13.5 扩展图层链路

```text
降雨/道路/历史易涝点数据 -> PostGIS 图层表 -> 图层 API -> 前端图层开关和图例
```

### 13.6 主线预测展示链路

```text
历史水位 + 当前趋势 + 降雨摘要 -> 规则型预测结果 -> 预测 API -> 地图弹窗和预测页面
```

## 14. 异常与降级规则

| 场景 | 处理方式 |
|---|---|
| 开放数据接口失败 | 保留最近一次成功数据，数据状态页显示失败原因 |
| 水位为空 | 保留原始记录，不参与风险计算，风险显示 `nodata` |
| 点位无审核坐标 | 不进入正式地图，保留在点位校核列表 |
| 高德地理编码失败 | 记录失败原因，允许人工录入经纬度 |
| 完全重复记录 | 不重复进入业务统计 |
| PostGIS 未就绪 | 一期 SQLite 闭环继续可用 |
| 降雨数据缺失 | 降雨图层显示暂无数据，不影响核心地图 |
| 道路数据缺失 | 道路图层显示暂无数据，不影响核心地图 |
| 历史易涝点缺失 | 易涝点图层显示暂无数据，不影响核心地图 |
| 告警、工单、审计未实现 | 不影响主线地图、统计、历史、校核、预测和数据状态交付 |
| 复杂预测数据不足 | 使用规则型或轻量预测，不影响主线验收 |
