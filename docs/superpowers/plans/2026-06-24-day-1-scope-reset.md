# Day 1 范围重置清单

> 更新基线：2026-06-24  
> 对应新排期：Day 1 范围重置与目标确认  
> 项目名称：深圳市积水监测与预测系统

## 1. Day 1 结论

项目目标调整为“监测与预测系统”，不是单纯地图展示，也不是一开始就训练复杂模型。正确推进顺序为：

```text
监测闭环
-> 坐标补全
-> 规则研判
-> 训练数据沉淀
-> 模型训练
-> 模型预测服务
-> 前端展示预测结果
```

## 2. 当前必须明确的事实

- 已有积涝点水位数据和测站基础信息。
- 当前测站基础信息缺少经纬度。
- 当前没有降雨量数据，第一版模型不能使用降雨特征。
- 高德 API 坐标可以作为临时坐标进入系统和初期训练，但必须记录来源和精度。
- 政府标准坐标需要同步申请，到位后校正样本并重新评估或重训模型。

## 3. 第一阶段目标：监测闭环

必须完成：

- 实时/定时水位接入。
- 原始 JSON/CSV 归档。
- SQLite 入库。
- 数据质量检查。
- 高德 API 初步坐标补全。
- 坐标来源、精度、审核状态记录。
- FastAPI 地图、历史、统计、数据状态接口。
- Vue + 高德地图展示。

验收标准：

- 数据能从开放数据或本地归档进入 SQLite。
- 地图可以展示有坐标的点位。
- 点位弹窗显示水位、风险、趋势、坐标来源。
- 数据状态页面显示最新导入时间和异常信息。

## 4. 第二阶段目标：规则研判

初期不称为模型预测，只做规则研判和趋势提示。

必须完成：

- 风险等级规则。
- 趋势判断规则。
- 数据新鲜度判断。
- 规则说明。
- 研判结果回写和前端展示。

输出字段：

- `risk_level`
- `trend`
- `rule_version`
- `rule_explanation`
- `generated_at`

## 5. 第三阶段目标：模型训练与预测

当历史样本足够后，再训练时序模型。

模型输入：

- 历史水位序列。
- 水位变化速度。
- 时间特征。
- 测站特征。
- 高德临时坐标及坐标来源。
- 后续政府标准坐标。
- 后续降雨数据。

模型输出：

- 未来水位或未来风险等级。
- 预测趋势。
- 置信度。
- 模型版本。

## 6. 数据划分原则

时序模型不能随机切分，应按预测目标时间切分。

建议第一版：

```text
训练集：2026-01-01 至 2026-04-30
验证集：2026-05-01 至 2026-05-31
测试集：2026-06-01 至 2026-06-23
```

## 7. 后置范围

不阻塞当前主线：

- 降雨数据。
- PostgreSQL + PostGIS。
- 后端调度平台化。
- 点位级阈值。
- 告警中心。
- 用户、角色、审计。
- 工单处置。
