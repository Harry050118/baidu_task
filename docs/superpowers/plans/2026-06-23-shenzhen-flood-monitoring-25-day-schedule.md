# 深圳市积水实时监测地图系统 25 天项目排期

> 排期口径：从 2026-06-23 开始，按 25 个自然日规划，结束日期为 2026-07-17。
> 最终目标：25 天内优先完成积水监测主线闭环和可演示版本；告警中心、用户角色审计、工单、预测展示等分支能力后置。
> 技术路线：FastAPI + Vue + 高德 JS API；前期用 SQLite 快速打通主线，PostgreSQL + PostGIS、调度、阈值和扩展图层作为后续增强。

## 1. 目标范围

本项目 25 天内以主线闭环为核心，不再把所有二期分支能力作为必须先完成内容。推进顺序调整为：

主线目标是完成基础可运行闭环：

- 开放数据采集和原始数据归档。
- SQLite 入库和数据质量检查。
- 点位坐标候选生成与人工校核。
- 统一风险等级和趋势计算。
- 高德地图展示积涝点、行政区边界和点位详情。
- FastAPI 地图、统计、历史、校核和数据状态接口。
- Vue + 高德地图指挥展示页。
- 全市统计、行政区统计、历史水位查询、点位校核和数据状态页面。

后续增强目标是在主线基础上逐步补强：

- SQLite 迁移到 PostgreSQL + PostGIS。
- 采集脚本升级为后端调度任务。
- 支持点位级风险阈值。
- 接入降雨数据。
- 接入道路和历史易涝点图层。

暂不优先完成的分支能力：

- 告警推送或站内告警中心。
- 用户、角色和审计日志。
- 处置工单。
- 预测模型结果展示。

25 天周期较紧，因此排期以“主线完整、可运行、可演示、可交付”为标准。分支能力只保留设计、接口位置或可选演示，不阻塞主线交付。真实短信网关、复杂机器学习训练、多部门联动和生产级高可用部署不作为本轮硬性目标。

## 5. 模块工作拆分

### 5.1 一期数据与地图闭环

目标：先让系统具备完整可运行主链路。

任务：

- 下载和归档开放数据。
- 导入 SQLite。
- 处理 NULL 水位和重复 ID 口径。
- 计算统一风险等级和趋势。
- 提供地图点位、统计、历史查询、坐标校核 API。
- 实现高德地图、态势面板、历史查询、点位校核和数据状态页面。

验收：

- 数据能从本地库进入后端 API。
- 地图能展示审核通过的积涝点和风险颜色。
- 点位详情能展示水位、风险、趋势和观测时间。
- 可以查询单点历史水位。
- 可以查看数据导入状态。

### 5.2 后续数据库与空间数据增强

目标：在主线闭环稳定后，再将 SQLite 闭环升级到 PostGIS 空间底座。该部分为后置增强，不阻塞当前主线验收。

任务：

- 建立 PostgreSQL + PostGIS。
- 迁移 SQLite 历史数据。
- 建立监测点、坐标、行政区、道路、历史易涝点、降雨和阈值表；告警、工单、用户审计、预测表只做预留设计。
- 创建空间索引和常用查询索引。
- 保留一期数据质量口径。

验收：

- PostGIS 中能查询监测点、水位历史、行政区、道路、易涝点和降雨数据。
- 地图点位只使用审核通过坐标。
- 业务计算排除 `water_level_m IS NULL`。
- 完全重复 ID 不重复进入业务统计。

### 5.3 后置后端增强

目标：在主线 API 稳定后，再提供调度、阈值、告警、权限、工单和预测 API。当前优先完成地图、统计、历史、校核和数据状态 API。

任务：

- 将采集脚本升级为后端调度任务。
- 实现点位级风险阈值。
- 实现降雨、道路、历史易涝点图层接口。
- 告警生成、用户角色审计、处置工单、预测结果查询先保留接口设计，不作为当前实现优先级。

验收：

- 调度任务可手动触发并记录状态。
- 主线 API 稳定可用。
- 后置分支接口边界清楚，不影响主线运行。

### 5.4 后置前端增强

目标：在一期指挥地图基础上预留扩展图层和分支业务入口。当前优先完成地图首页、历史查询、点位校核和数据状态页面。

任务：

- 展示降雨、道路、历史易涝点图层。
- 实现图层开关和图例。
- 告警中心、阈值配置、用户审计、调度状态、处置工单和预测结果页面先保留设计，不先实现。

验收：

- 指挥地图能同时展示核心监测点和扩展图层。
- 主线页面可演示。
- 后置分支入口和页面设计清楚，不阻塞主线交付。
